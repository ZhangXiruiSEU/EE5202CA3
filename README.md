# Zephyr vs STM32CubeIDE In This Project

This assignment is commonly done with STM32CubeIDE, STM32CubeMX, HAL drivers,
and generated `Core/Src/main.c` / `Core/Inc` project structure. This project
does not use that workflow. It is a Zephyr application built with CMake and
Zephyr's device, sensor, kernel, and devicetree infrastructure.

The target board for the current build is:

```text
disco_l475_iot1
```

Zephyr resolves that board to its built-in board support directory:

```text
zephyr/boards/st/disco_l475_iot1
```

Because Zephyr already supports this board, the project does not need to write
the basic hardware interface layer for I2C, SPI, UART, clocks, GPIO routing, or
the onboard sensor descriptions. Those are provided by Zephyr's board
definition and devicetree files.

## Development Environment

This project assumes a Zephyr command-line workspace rather than a standalone
CubeIDE workspace.

Set up Zephyr by following the official Getting Started Guide:

```text
https://docs.zephyrproject.org/latest/develop/getting_started/index.html
```

After the Zephyr workspace is ready, place this project folder inside the
workspace, for example:

```text
~/zephyrproject/
  zephyr/
  applications/
    lsm6dsl/
```

The application directory would then be:

```text
~/zephyrproject/applications/lsm6dsl
```

The exact workspace path does not matter; the commands below assume the example
`~/zephyrproject` layout.

## Build And Flash

From the application directory:

```sh
cd ~/zephyrproject/applications/lsm6dsl
```

Build with `west`:

```sh
west build -b disco_l475_iot1
```

Flash to the connected board:

```sh
west flash
```

## Build System

### STM32CubeIDE Default Style

A typical CubeIDE project is generated by CubeMX and built as an IDE-managed
STM32 project. The project usually contains files such as:

```text
.ioc
Core/Src/main.c
Core/Src/stm32xxxx_hal_msp.c
Core/Src/stm32xxxx_it.c
Core/Inc/main.h
Drivers/STM32xxxx_HAL_Driver
Drivers/CMSIS
```

### This Project

This project keeps the application files in a small Zephyr application
structure:

```text
lsm6dsl/
  CMakeLists.txt
  prj.conf
  src/
    main.c
    app_threads.c
    sensor_tasks.c
    status.c
    sensor_drdy.c
    lsm6dsl_fifo.c
    HARnet/
      network.c
      network_data.c
      Lib/NetworkRuntime1200_CM4_GCC.a
```

`CMakeLists.txt` is the application build description. It lists the C source
files and links the HAR static library.

`prj.conf` is the Zephyr configuration file for this application. It enables
the drivers and kernel features used by the code.

`src/` contains the application logic.

The board support, MCU definition, bus
configuration, and onboard sensor descriptions are not stored in this folder;
they come from Zephyr's `disco_l475_iot1` board files.

`build/` is generated by `west build`. It contains CMake output, generated
headers, object files, and the final firmware image, so it is not part of the
source structure.


## Configuration

### STM32CubeIDE Default Style

CubeIDE uses CubeMX-generated initialization code and an `.ioc` file to define
peripheral setup.

For example, I2C, UART, clocks, GPIOs, and interrupts are configured in CubeMX,
then emitted as HAL initialization code.

### This Project

Zephyr uses Kconfig and devicetree.

The application configuration is in `prj.conf`:

```text
CONFIG_STDOUT_CONSOLE=y
CONFIG_I2C=y
CONFIG_SPI=y
CONFIG_SENSOR=y
CONFIG_FPU=y
CONFIG_FP_HARDABI=y
CONFIG_LSM6DSL_TRIGGER_NONE=y
CONFIG_HTS221_ODR="1"
CONFIG_LIS3MDL_ODR="1.25"
CONFIG_LPS22HB_SAMPLING_RATE=1
CONFIG_CBPRINTF_FP_SUPPORT=y
CONFIG_MAIN_STACK_SIZE=6144
```

This means the sensor stack, I2C/SPI support, console output, FPU behavior, and
sensor default ODRs are enabled through Zephyr configuration instead of CubeMX
code generation.

## Main Entry

### STM32CubeIDE Default Style

CubeIDE usually has a generated `main()` like:

```c
int main(void)
{
    HAL_Init();
    SystemClock_Config();

    MX_GPIO_Init();
    MX_I2C1_Init();
    MX_USART2_UART_Init();

    while (1) {
    }
}
```

### This Project

The current Zephyr `main()` does not call `HAL_Init()`,
`SystemClock_Config()`, or CubeMX-generated `MX_*_Init()` functions. Zephyr
starts the kernel and runs its configured system and device initialization
before application `main()` is entered.

The application still checks that the devices it needs are ready with:

```c
device_is_ready(ctx->lsm6dsl_dev)
```

Then it performs project-specific initialization, such as HAR setup, DRDY
diagnostics, and LSM6DSL FIFO configuration, before starting application
threads:

```c
app_threads_start(&app);
```

Finally, the main thread sleeps forever:

```c
while (1) {
    k_sleep(K_FOREVER);
}
```

This is Zephyr style: the application work runs in kernel threads rather than a
single CubeIDE `while (1)` loop.

## Driver Access

### STM32CubeIDE Default Style

In CubeIDE, sensor drivers are often accessed through HAL I2C/SPI calls:

```c
HAL_I2C_Mem_Read(...);
HAL_I2C_Mem_Write(...);
HAL_SPI_TransmitReceive(...);
```

Or through ST BSP sensor wrappers if available.

### This Project

Zephyr's board devicetree describes the onboard sensors and their bus
connections. For this board, the relevant file is:

```text
zephyr/boards/st/disco_l475_iot1/disco_l475_iot1.dts
```

A shortened excerpt is:

```dts
/ {
	model = "STMicroelectronics B-L475E-IOT01Ax board";
	compatible = "st,stm32l475-disco-iot";

	chosen {
		zephyr,console = &usart1;
		zephyr,sram = &sram0;
		zephyr,flash = &flash0;
	};

	aliases {
		accel0 = &lsm6dsl;
	};
};

&i2c2 {
	pinctrl-0 = <&i2c2_scl_pb10 &i2c2_sda_pb11>;
	pinctrl-names = "default";
	status = "okay";
	clock-frequency = <I2C_BITRATE_FAST>;

	lis3mdl-magn@1e {
		compatible = "st,lis3mdl-magn";
		reg = <0x1e>;
	};

	hts221@5f {
		compatible = "st,hts221";
		reg = <0x5f>;
		drdy-gpios = <&gpiod 15 GPIO_ACTIVE_HIGH>;
	};

	lps22hb-press@5d {
		compatible = "st,lps22hb-press";
		reg = <0x5d>;
	};

	lsm6dsl: lsm6dsl@6a {
		compatible = "st,lsm6dsl";
		reg = <0x6a>;
		irq-gpios = <&gpiod 11 GPIO_ACTIVE_HIGH>;
	};
};
```

This is why the application can get sensor device objects by compatible name:

```c
DEVICE_DT_GET_ONE(st_lsm6dsl)
DEVICE_DT_GET_ONE(st_hts221)
DEVICE_DT_GET_ONE(st_lis3mdl_magn)
DEVICE_DT_GET_ONE(st_lps22hb_press)
```

instead of manually defining I2C addresses, pins, and HAL handles in application
code.

Most sensor reads use Zephyr's generic sensor API:

```c
sensor_sample_fetch(...)
sensor_sample_fetch_chan(...)
sensor_channel_get(...)
sensor_attr_set(...)
```

Examples in this project:

```c
sensor_sample_fetch_chan(ctx->lsm6dsl_dev, SENSOR_CHAN_GYRO_XYZ);
sensor_channel_get(ctx->lsm6dsl_dev, SENSOR_CHAN_GYRO_XYZ, gyro);
```

This makes the code depend on Zephyr's sensor abstraction rather than direct HAL
register access.

The DRDY helper and FIFO helper are exceptions where the project reads sensor
registers directly through Zephyr I2C:

```c
i2c_reg_read_byte_dt(...)
i2c_reg_write_byte_dt(...)
i2c_burst_read_dt(...)
```

Even there, the bus access still goes through Zephyr's `i2c_dt_spec`, not HAL
`I2C_HandleTypeDef`.

## Timing And Scheduling

### STM32CubeIDE Default Style

CubeIDE/HAL projects often use:

```c
HAL_GetTick()
HAL_Delay()
TIM interrupts
SysTick
FreeRTOS, if enabled
```

### This Project

This project uses Zephyr kernel timing:

```c
k_sleep(...)
k_uptime_ticks()
k_ticks_to_us_floor64(...)
k_cycle_get_32()
k_cyc_to_us_floor64(...)
```

The current multithreaded branch uses Zephyr threads:

```c
k_thread_create(...)
K_THREAD_STACK_DEFINE(...)
```

It also uses Zephyr synchronization and queues:

```c
K_MUTEX_DEFINE(app_lock)
K_MUTEX_DEFINE(sensor_lock)
K_MSGQ_DEFINE(accel_msgq, ...)
k_mutex_lock(...)
k_msgq_put(...)
k_msgq_get(...)
```

This is a major difference from a basic CubeIDE superloop.

## Console Output

### STM32CubeIDE Default Style

CubeIDE projects often redirect `printf()` by implementing `_write()` and
calling:

```c
HAL_UART_Transmit(...)
```

### This Project

This project uses Zephyr console output:

```c
printk(...)
```

String formatting uses Zephyr's `vsnprintk()`:

```c
vsnprintk(...)
```

Console behavior is enabled through:

```text
CONFIG_STDOUT_CONSOLE=y
CONFIG_CBPRINTF_FP_SUPPORT=y
```

So UART / console output is managed by Zephyr rather than direct HAL UART code.

## HAR Network Integration

The HAR model comes from ST AI generated code and a prebuilt runtime library:

```text
src/HARnet/network.c
src/HARnet/network_data.c
src/HARnet/Inc
src/HARnet/Lib/NetworkRuntime1200_CM4_GCC.a
```

This part is relatively portable between Zephyr and CubeIDE because it is plain
C plus a static library, but the build integration differs.

In Zephyr CMake, it is linked with:

```cmake
target_link_libraries(app PRIVATE
  ${CMAKE_CURRENT_SOURCE_DIR}/src/HARnet/Lib/NetworkRuntime1200_CM4_GCC.a
)
```

In CubeIDE, the same library would need to be added through project linker
settings, and include paths would need to be configured in the IDE.

## Zephyr And CubeIDE Comparison

The difference is not the C language itself. The main difference is which layer
owns board setup, driver access, scheduling, and build configuration.

| Area | This Zephyr project | STM32CubeIDE default approach |
| --- | --- | --- |
| Board description | Uses Zephyr's built-in `disco_l475_iot1` board files and devicetree. | Uses CubeMX `.ioc` configuration and generated initialization code. |
| Device access | Gets devices with `DEVICE_DT_GET_ONE(...)` and checks them with `device_is_ready()`. | Uses generated HAL handles such as `I2C_HandleTypeDef` or BSP-specific sensor handles. |
| Sensor reads | Uses Zephyr sensor APIs such as `sensor_sample_fetch()` and `sensor_channel_get()`. | Usually uses ST sensor drivers, BSP functions, or direct HAL I2C/SPI register reads. |
| Scheduling | Uses Zephyr threads, priorities, sleeps, mutexes, and message queues. | Usually uses a superloop, HAL timers, interrupts, or FreeRTOS if enabled manually. |
| Configuration | Enables drivers and features in `prj.conf`. | Enables peripherals and middleware through CubeMX and project settings. |
| Build system | Uses Zephyr CMake through `west build`. | Uses CubeIDE managed project settings and generated makefiles. |
| Console output | Uses Zephyr `printk()` and console configuration. | Usually redirects `printf()` to UART with HAL transmit code. |
| HAR model | Links ST AI generated C code and a static runtime library through CMake. | Would add the same generated code and library through CubeIDE include/linker settings. |

So the project logic can be similar in both environments, but the surrounding
infrastructure is different. Zephyr provides board support, drivers, RTOS
objects, and build integration as reusable framework pieces; CubeIDE normally
generates a board-specific HAL project where the application owns more of the
peripheral setup directly.
